# 一、模块概述

## 1.1 模块名称

App 接入管理模块（App Access Management Module）

## 1.2 模块定位

作为游戏智能化营销推送系统的**多游戏App统一接入入口与基础支撑模块**，承担游戏客户端与推送系统的标准化对接、App信息全生命周期管理、设备令牌（Device Token/FCM Token）采集与维护、接入鉴权与多App隔离、推送渠道配置等核心职责，是系统实现多租户管理、精准圈人、高效Push下发的基础前提，直接支撑圈人模块、推送模块、数据统计模块的正常运行。

## 1.3 建设目标

- 标准化：制定统一的App接入规范与接口标准，支持单款/多款游戏App快速接入，降低游戏研发对接成本，实现接入流程标准化、可复用。

- 安全性：建立完善的接入鉴权机制（AK/SK、IP白名单），确保接口调用安全，防止非法接入与数据泄露。

- 隔离性：实现多游戏App的数据、权限、推送完全隔离，避免跨App串推、数据混淆，保障多游戏并行运营需求。

- 可靠性：完成玩家设备Token（FCM/APNs）的实时采集、同步、失效清理，确保Push下发的准确性，减少无效推送。

- 可配置：支持按App维度灵活配置推送渠道（FCM为主）、频控、黑名单、运营权限等，适配不同游戏的个性化推送需求。

- 可支撑：为后续人群圈选（基于App下设备/玩家）、Push下发（基于App渠道配置）、数据统计（按App维度拆分）提供完整的基础数据支撑。

## 1.4 适用范围

本PRD适用于游戏智能化营销推送系统中，App接入模块的需求定义、开发、测试、上线、运维全流程，覆盖系统管理员、游戏研发、游戏运营、测试人员等所有相关角色，明确模块功能边界、交互逻辑、非功能要求及与其他模块的关联关系。

## 1.5 模块边界

### 1.5.1 负责范围

- App基础信息（AppID、包名等）的新增、编辑、启用、禁用、查询、删除全生命周期管理。

- 接入鉴权（AK/SK生成、重置、禁用、验签）与IP白名单配置。

- 推送渠道（FCM/APNs等）的按App独立配置与管理。

- 设备信息（Device Token、设备型号等）的上报、更新、去重、失效清理与查询。

- App接入审核流程的管理（提交、审核、退回、通过）。

- 多App的数据、权限、推送隔离机制的实现。

- App级运营辅助配置（黑名单、推送开关、测试白名单等）。

- 接入相关操作日志的记录与查询。

### 1.5.2 不负责范围

- Push消息的具体下发（由推送模块负责）。

- 人群圈选的具体逻辑（由圈人模块负责）。

- 推送数据的统计分析（由数据统计模块负责）。

- 用户权限的细粒度分配（由权限模块负责，本模块仅关联权限分配结果）。

- 游戏客户端的开发与调试（仅提供接入文档与调试支持）。

## 2.2 典型使用场景（详细场景）

### 场景1：新游戏App接入（核心场景）

1. 前置条件：新游戏研发完成客户端开发，获取FCM项目配置（Server Key/ServiceAccountJson）、App包名等信息。

2. 操作流程：
        

    - 游戏研发登录系统，进入App接入管理页面，点击“新增App”，填写App基础信息（游戏名称、包名、平台类型等），上传FCM配置文件，提交接入申请。

    - 系统管理员收到审核通知，进入审核页面，查看接入资料的完整性与准确性（包名是否正确、FCM配置是否有效等）。

    - 审核通过：系统自动生成该App的AK/SK，设置App状态为“已上线”，并通知游戏研发。

    - 审核拒绝：系统管理员填写拒绝原因（如FCM配置无效、包名错误），将申请退回，游戏研发修改后重新提交。

    - 游戏研发获取AK/SK后，集成设备上报接口，在游戏客户端实现Token上报功能，调试接口调用是否正常。

    - 调试通过后，游戏正式上线，设备Token开始实时上报至推送系统，为后续推送做准备。

3. 异常场景：研发提交的FCM配置无效，审核被退回；接口调试失败，研发查询操作日志排查问题。

### 场景2：玩家设备Token上报与更新

1. 前置条件：游戏App已接入推送系统，玩家安装并启动游戏，客户端已集成Token上报接口。

2. 操作流程：
        

    - 玩家启动游戏，客户端自动获取设备的FCM Token（Android）/APNs Token（iOS），携带玩家ID、AppID、设备信息等参数，调用Token上报接口。

    - 推送系统接口校验AK/SK有效性、参数完整性，通过后执行以下操作：
                

        - 若该设备为首次上报，新增设备记录，关联玩家ID与AppID。

        - 若该设备已上报（按设备ID匹配），更新Token（如Token变更）、最近活跃时间、系统版本等信息。

        - 若该玩家关联多台设备，自动关联所有设备记录，形成玩家-设备映射关系。

    - 接口返回上报成功状态，客户端完成上报流程。

    - 系统定时（每日）清理N天（可配置，默认30天）未活跃的设备，标记为“失效”，避免无效推送。

3. 异常场景：Token上报参数缺失、AK/SK错误，接口返回失败；Token格式错误，系统拒绝接收。

### 场景3：App推送渠道配置修改

1. 前置条件：游戏App已上线，因FCM项目更新、推送策略调整，需要修改推送渠道配置。

2. 操作流程：
        

    - 系统管理员登录系统，进入App接入管理列表，找到目标App，点击“渠道配置”。

    - 修改FCM配置（Server Key/ServiceAccountJson）、推送环境（生产/测试）、失败补发策略、频控上限等参数。

    - 点击“保存”，系统校验配置有效性（如FCM密钥是否可用），校验通过后生效；校验失败，提示错误信息，无法保存。

    - 系统记录配置修改日志（修改人、修改时间、修改前后内容），便于追溯。

    - 游戏运营查看渠道配置，确认修改内容，后续Push下发将使用新的渠道配置。

3. 异常场景：修改后的FCM密钥无效，系统校验失败；修改后未保存，配置未生效。

### 场景4：多游戏App隔离与权限管理

1. 前置条件：系统已接入多款游戏App，需要实现多App隔离，分配不同运营账号的操作权限。

2. 操作流程：
        

    - 系统管理员在权限模块为游戏运营账号分配对应App的操作权限（如仅允许操作“游戏A”）。

    - 游戏运营登录系统，进入App接入管理页面，仅能看到自己有权限的App（游戏A），无法看到其他App（游戏B、游戏C）的信息。

    - 游戏运营操作游戏A的黑名单、测试白名单时，仅对游戏A生效，不影响其他App。

    - 推送系统下发Push时，仅读取对应App的渠道配置与设备列表，确保推送范围不跨App，避免串推。

    - 数据统计模块按App维度拆分数据，游戏运营仅能查看游戏A的设备量、推送数据，无法查看其他App的数据。

3. 异常场景：权限分配错误，运营账号能看到无权限的App；推送系统出现Bug，导致跨App串推（需触发告警）。

### 场景5：Token失效清理与手动剔除

1. 前置条件：部分玩家卸载游戏、更换设备，导致Token失效，推送系统收到FCM回执提示Token无效。

2. 操作流程：
        

    - 推送系统接收FCM返回的Token失效回执，调用本模块的Token失效标记接口，将对应Token标记为“失效”。

    - 系统定时（每日）执行失效Token清理任务，删除或归档N天（可配置）未活跃且标记为“失效”的设备记录。

    - 游戏运营发现部分无效设备导致推送失败率过高，可手动进入设备列表，筛选无效设备，执行“手动剔除”操作，立即标记为失效并清理。

    - 系统记录Token失效与清理日志，便于运营查看失效原因（如卸载、Token过期）。

3. 异常场景：FCM回执延迟，导致Token失效未及时标记；手动剔除错误，误删有效设备。

# 三、功能需求（详细版）

## 3.1 App基础信息管理（核心功能）

### 3.1.1 功能描述

实现游戏App基础信息的新增、编辑、启用、禁用、查询、删除全生命周期管理，确保每款App有唯一标识，信息准确可追溯。

### 3.1.2 核心字段定义（详细）

|字段名称|字段类型|是否必填|默认值|取值范围|字段说明|
|---|---|---|---|---|---|
|AppID|字符串|是|系统自动生成|长度16-32位，由字母、数字、下划线组成，唯一不重复|游戏App的唯一标识，用于接口调用、数据隔离、推送下发的核心关联字段，生成后不可修改|
|游戏名称|字符串|是|无|长度1-50位，不包含特殊字符|游戏的正式名称，用于前端展示、搜索、运营区分|
|包名（Package Name）|字符串|是|无|符合Android/iOS包名规范（如com.xxx.game）|Android端包名，iOS端填写Bundle ID，用于匹配设备Token所属App，不可重复|
|平台类型|单选|是|Android|Android、iOS、双平台|标识App支持的设备平台，用于后续渠道配置（如iOS需配置APNs）|
|App状态|单选|是|草稿|草稿、审核中、已上线、已禁用|草稿：未提交审核；审核中：等待管理员审核；已上线：可正常接收Token、推送；已禁用：禁止Token上报、推送下发|
|负责人|字符串|是|无|长度1-50位|App的负责人（游戏研发/运营），用于对接、通知相关事宜|
|负责人联系方式|字符串|否|无|手机号/邮箱|负责人的联系方式，便于紧急对接|
|备注|文本|否|无|长度0-200位|记录App相关备注信息（如上线时间、特殊需求等）|
|创建时间|时间戳|是|系统当前时间|格式：yyyy-MM-dd HH:mm:ss|App信息创建的时间，不可修改|
|更新时间|时间戳|是|系统当前时间|格式：yyyy-MM-dd HH:mm:ss|App信息最后一次更新的时间，修改信息后自动更新|
|删除标识|布尔值|是|false|true（已删除）、false（未删除）|逻辑删除标识，删除App时仅标记为true，不物理删除数据|
### 3.1.3 具体操作功能

- 新增App：
        

    - 点击“新增App”按钮，弹出新增表单，填写上述必填字段，可选字段可选择性填写。

    - 表单校验：包名需符合规范、不可重复；游戏名称不可为空；负责人不可为空。

    - 提交后，App状态自动设为“草稿”，可继续编辑或提交审核。

- 编辑App：
        

    - 仅可编辑“草稿”“审核拒绝”状态的App；“审核中”“已上线”“已禁用”状态的App，仅可编辑备注、负责人、联系方式，不可编辑AppID、包名、平台类型。

    - 编辑后点击“保存”，更新时间自动更新，表单校验规则与新增一致。

- 提交审核：
        

    - 仅“草稿”状态的App可提交审核，提交后状态变为“审核中”，不可再编辑。

    - 提交时需确认App信息与FCM配置（若已填写）的完整性，提示“确认提交审核？提交后不可编辑”。

- 审核操作（管理员）：
        

    - 审核通过：点击“审核通过”，系统自动生成AK/SK，App状态变为“已上线”，并发送通知给负责人。

    - 审核拒绝：点击“审核拒绝”，填写拒绝原因（必填），App状态变为“审核拒绝”，并发送通知给负责人，允许负责人修改后重新提交。

- 启用/禁用App：
        

    - 仅“已上线”状态的App可禁用，禁用后状态变为“已禁用”，禁止Token上报、推送下发。

    - 仅“已禁用”状态的App可启用，启用后状态变为“已上线”，恢复Token上报、推送下发功能。

    - 操作时需确认，提示“确认禁用该App？禁用后将停止所有推送与Token上报”。

- 查询App：
        

    - 支持按游戏名称（模糊搜索）、AppID（精确搜索）、包名（模糊搜索）、平台类型（下拉筛选）、App状态（下拉筛选）进行搜索。

    - 搜索结果展示AppID、游戏名称、包名、平台类型、状态、负责人、创建时间，支持分页（每页10/20/50条）。

- 删除App：
        

    - 仅“草稿”“审核拒绝”状态的App可删除，删除后逻辑删除标识设为true，前端不再展示，数据保留在数据库中可追溯。

    - 操作时需确认，提示“确认删除该App？删除后不可恢复”。

## 3.2 接入鉴权与安全（核心功能）

### 3.2.1 功能描述

为每款App分配唯一的AccessKey（AK）与SecretKey（SK），实现接口调用的身份验证；支持IP白名单配置，限制合法调用IP；记录所有鉴权相关操作日志，确保接入安全。

### 3.2.2 核心字段定义

|字段名称|字段类型|是否必填|默认值|字段说明|
|---|---|---|---|---|
|AccessKey（AK）|字符串|是|系统自动生成|长度32位，由字母、数字随机组成，唯一不重复，用于接口调用时的身份标识，可公开|
|SecretKey（SK）|字符串|是|系统自动生成|长度64位，由字母、数字、特殊字符随机组成，唯一不重复，用于接口验签，不可公开，仅管理员与负责人可查看|
|AK/SK状态|单选|是|启用|启用（可正常使用）、禁用（不可使用），用于临时关闭接口调用权限|
|IP白名单|文本（多行）|否|无|每行一个IP，支持网段（如192.168.1.0/24），为空时允许所有IP调用接口，不为空时仅允许白名单内IP调用|
|生成时间|时间戳|是|系统当前时间|AK/SK生成的时间，不可修改|
|重置时间|时间戳|否|无|AK/SK最后一次重置的时间，未重置则为空|
|重置人|字符串|否|无|最后一次重置AK/SK的用户，未重置则为空|
### 3.2.3 具体操作功能

- AK/SK生成：
        

    - 仅当App审核通过（状态变为“已上线”）时，系统自动生成AK/SK，生成后不可修改，仅可重置。

    - SK默认隐藏，点击“查看SK”可显示，显示时需二次确认（输入管理员密码/验证码），防止泄露。

- AK/SK重置：
        

    - 仅系统管理员可操作，点击“重置AK/SK”，系统生成新的AK/SK，旧的AK/SK立即失效。

    - 重置时需确认，提示“确认重置该App的AK/SK？重置后旧密钥立即失效，请及时更新游戏客户端配置”。

    - 重置后，系统记录重置日志（重置人、重置时间、旧AK），并通知App负责人。

- AK/SK启用/禁用：
        

    - 仅系统管理员可操作，点击“禁用”，AK/SK状态变为“禁用”，接口调用时验签失败；点击“启用”，状态恢复为“启用”。

    - 禁用时需确认，提示“确认禁用该App的AK/SK？禁用后游戏客户端无法调用接口，影响Token上报与推送”。

- IP白名单配置：
       

    - 系统管理员可编辑IP白名单，支持手动输入、批量粘贴（每行一个IP/网段）。

    - 输入时自动校验IP/网段格式，格式错误时提示“IP格式不正确，请重新输入”。

    - 保存后立即生效，接口调用时会校验调用IP是否在白名单内（为空时不校验）。

- 鉴权逻辑：
        

    - 游戏客户端调用接口时，需在请求头携带AK、签名（由SK与请求参数加密生成）、请求时间戳。

    - 系统校验：AK是否存在且启用 → IP是否在白名单内（若配置） → 签名是否正确 → 时间戳是否在有效范围内（默认5分钟，防止重放攻击）。

    - 校验通过：正常处理接口请求；校验失败：返回对应错误码（如AK无效、签名错误、IP不在白名单），拒绝处理请求。

- 鉴权日志：
        

    - 记录所有接口调用的鉴权情况，包括调用IP、AK、请求接口、请求时间、鉴权结果（成功/失败）、失败原因。

    - 支持按AK、调用IP、鉴权结果、时间范围搜索日志，便于排查接口调用失败问题。

### 3.3.3 具体操作功能

- 渠道配置编辑：
        

    - 系统管理员进入App详情页，点击“渠道配置”，进入配置页面，根据App平台类型填写对应渠道配置（Android填写FCM，iOS填写APNs，双平台均填写）。

    - FCM配置：支持上传ServiceAccountJson文件，或手动输入Server Key与项目ID，二选一即可；系统自动校验配置有效性（调用FCM测试接口，确认密钥可用）。

    - APNs配置：支持上传.p12格式证书，输入证书密码，系统自动校验证书有效性；推送环境需与证书环境一致（测试证书对应测试环境，生产证书对应生产环境）。

    - 渠道优先级：支持拖拽排序，可调整FCM、站内信、短信的优先级顺序，排序后立即生效。

    - 频控配置：可修改单机日推送上限、单用户频控上限，输入时需校验数值范围（1-100条/1-10条/小时）。

    - 保存配置：点击“保存”，系统校验配置完整性与有效性，校验通过后生效；校验失败，提示具体错误信息（如FCM密钥无效、APNs证书格式错误），无法保存。

- 配置启用/禁用：
        

    - 系统管理员可点击“启用/禁用”按钮，切换渠道配置状态；禁用时，该App无法下发Push，需确认提示“确认禁用该App的推送渠道配置？禁用后将无法下发任何Push”。

- 配置测试：
        

    - 配置完成后，系统管理员可点击“测试推送”，输入测试设备Token、推送标题、内容，选择推送渠道，发送测试Push，验证配置是否有效。

    - 测试结果实时反馈（推送成功/失败），失败时提示失败原因（如Token无效、渠道配置错误），便于调试。

- 配置日志：

    - 记录所有渠道配置的修改操作，包括修改人、修改时间、修改前后的配置内容，便于追溯。
