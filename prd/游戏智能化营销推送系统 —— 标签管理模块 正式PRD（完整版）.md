# 一、模块概述

## 1.1 模块名称

标签管理模块（Tag Management Module）

## 1.2 模块定位

作为游戏智能化营销推送系统的**核心支撑模块**，衔接设备信息管理模块、圈人模块、推送模块，承担标签全生命周期管理、标签与设备/玩家关联、标签规则配置、标签权限隔离等核心职责，建立标准化、可扩展的标签体系。通过对玩家设备、行为、偏好等数据的标签化管理，为圈人模块提供精准的标签筛选能力，为推送模块提供个性化推送的标签依据，是实现“精准圈人、高效推送”的核心基础，直接支撑系统营销转化效率的提升。

## 1.3 建设目标

- 标准化：制定统一的标签分类、命名、创建规范，建立覆盖设备、玩家、行为的标准化标签体系，确保标签的统一性和可复用性，适配多游戏并行运营需求。

- 灵活性：支持系统标签自动生成、自定义标签手动创建，支持标签规则灵活配置、动态更新，适配不同游戏的个性化标签需求（如不同游戏的玩家行为标签差异）。

- 准确性：实现标签与设备、玩家的精准关联，支持标签数据实时更新、失效清理，确保标签数据的准确性和时效性，避免因标签错误导致推送偏差。

- 隔离性：实现多游戏App标签数据、操作权限的完全隔离，避免跨App标签混淆、权限越界，保障多游戏独立运营。

- 可支撑：为圈人模块提供完整的标签筛选能力（按标签组合圈选人群），为推送模块提供标签匹配能力（按标签定向推送），为数据统计模块提供标签维度的数据分析支撑。

- 可追溯：记录所有标签相关操作日志（创建、编辑、关联、删除等），确保标签操作可追溯、可审计，便于问题排查和运营复盘。

## 1.4 适用范围

本PRD适用于游戏智能化营销推送系统中，标签管理模块的需求定义、开发、测试、上线、运维全流程，覆盖系统管理员、游戏研发、游戏运营、测试人员等所有相关角色，明确模块功能边界、交互逻辑、非功能要求及与其他模块的关联关系，作为模块开发、测试验收的核心依据。

## 1.5 模块边界

### 1.5.1 负责范围

- 标签基础信息（标签ID、名称、类型、描述等）的新增、编辑、启用、禁用、查询、删除全生命周期管理。

- 标签分类管理，包括分类的新增、编辑、删除，标签与分类的关联配置。

- 标签与设备、玩家的关联管理，包括自动关联（基于规则）、手动关联、关联关系更新、解除与清理。

- 标签规则配置，包括系统标签生成规则、自定义标签关联规则的配置、生效、失效管理。

- 标签权限管理，按角色、按App维度分配标签操作权限，实现多App标签隔离。

- 标签操作日志的记录、查询与导出，支持按多种条件筛选日志。

- 标签数据统计，包括标签关联设备数、玩家数、标签使用频率等基础数据统计。

- 标签数据的实时同步，向圈人模块、推送模块提供标签数据支撑。

### 1.5.2 不负责范围

- 玩家行为数据的采集（由设备信息管理模块、游戏客户端负责）。

- 基于标签的人群圈选具体逻辑（由圈人模块负责）。

- 按标签定向推送的具体下发逻辑（由推送模块负责）。

- 标签维度的深度数据分析（如标签转化率、标签人群画像）（由数据统计模块负责）。

- 用户权限的细粒度分配（由权限模块负责，本模块仅关联权限分配结果）。

## 2.2 典型使用场景（详细场景）

### 场景1：系统标签规则配置与自动关联（核心场景）

1. 前置条件：系统已接入游戏App，设备信息管理模块可正常采集玩家设备、行为数据（如登录频率、消费金额、游戏时长）。

2. 操作流程：
        

3. 系统管理员登录系统，进入标签管理模块，点击“标签规则配置”，选择目标游戏App，进入系统标签规则配置页面。

4. 配置系统标签规则（如“高频登录用户”：近7天登录≥5次；“高消费用户”：累计消费≥100元），设置规则生效时间、更新频率（如每日凌晨更新）。

5. 点击“保存并生效”，系统校验规则合法性（如条件逻辑正确、数据字段存在），校验通过后规则生效。

6. 系统按规则更新频率，自动读取设备信息管理模块的玩家行为数据，匹配规则条件，将符合条件的玩家/设备与对应系统标签关联。

7. 游戏运营查看标签关联数据，确认标签自动关联的准确性，若发现异常，反馈给系统管理员调整规则。

8. 异常场景：规则条件设置错误（如字段不存在），系统校验失败；玩家行为数据缺失，导致标签关联失败；规则更新延迟，导致标签数据不及时。

### 场景2：自定义标签创建与手动关联

1. 前置条件：游戏运营已获取本游戏App的操作权限，明确自定义标签的用途（如“测试服玩家”“活动参与用户”）。

2. 操作流程：
        

3. 游戏运营登录系统，进入标签管理模块，切换至本游戏App，点击“新增标签”，选择标签类型为“自定义标签”。

4. 填写标签基础信息（名称、描述、分类），设置标签状态为“启用”，点击“保存”，系统校验标签名称唯一性（本App内不重复），校验通过后生成自定义标签。

5. 点击该标签的“手动关联”按钮，选择关联类型（设备/玩家），支持手动输入、批量粘贴设备ID/玩家ID，点击“确认关联”。

6. 系统校验设备ID/玩家ID的有效性（是否属于本App），校验通过后完成关联，更新标签关联数据统计。

7. 游戏运营查看标签关联列表，确认关联成功，后续可用于圈选人群、定向推送。

8. 异常场景：标签名称与本App内现有标签重复，保存失败；输入的设备ID/玩家ID无效或不属于本App，关联失败；批量粘贴格式错误（如多行重复、非有效ID），校验失败。

### 场景3：标签查询与筛选（支撑圈人场景）

1. 前置条件：游戏运营需筹备推送活动，需筛选符合特定条件的标签，用于圈选目标人群。

2. 操作流程：
        

3. 游戏运营登录系统，进入标签管理模块，切换至本游戏App，点击“标签查询”。

4. 设置筛选条件：标签分类、标签类型（系统/自定义）、标签状态（启用/禁用）、关联数量范围，支持标签名称模糊搜索。

5. 点击“查询”，系统展示符合条件的标签列表，包含标签ID、名称、类型、分类、关联设备数/玩家数、创建时间、状态。

6. 游戏运营点击目标标签，查看标签详情（关联的设备/玩家列表、操作日志），确认标签符合需求后，将标签同步至圈人模块，用于人群圈选。

7. 异常场景：筛选条件设置错误，无符合条件的标签；标签关联数据未及时更新，展示的关联数量与实际不符。

### 场景4：标签更新与失效处理

1. 前置条件：因游戏运营策略调整，某自定义标签（如“春节活动用户”）已失效，或某系统标签规则需优化（如“高频登录用户”的登录次数调整）。

2. 操作流程：
        

3. 若为自定义标签失效：游戏运营进入标签详情页，点击“禁用”，系统提示“确认禁用该标签？禁用后该标签不可用于圈选、推送，关联关系保留”，确认后标签状态变为“禁用”。

4. 若为系统标签规则优化：系统管理员进入标签规则配置页面，找到目标标签规则，修改规则条件（如将“近7天登录≥5次”改为“近7天登录≥4次”），点击“保存并生效”，系统自动更新标签关联数据。

5. 若标签无需保留：游戏运营（自定义标签）/系统管理员（系统标签）点击“删除”，确认后逻辑删除标签（关联关系归档，标签不再展示）。

6. 系统记录标签更新、禁用、删除操作日志，便于后续追溯。

7. 异常场景：误操作禁用/删除有效标签；系统标签规则修改后，关联数据更新失败，导致标签与实际玩家行为不匹配。

### 场景5：多游戏标签隔离与权限管理

1. 前置条件：系统已接入多款游戏App，需实现不同游戏标签的隔离，避免运营人员操作跨App标签。

2. 操作流程：
        

3. 系统管理员在权限模块，为不同游戏的运营账号分配对应App的标签操作权限（如游戏A运营仅能操作游戏A的标签）。

4. 游戏A运营登录系统，进入标签管理模块，仅能看到游戏A的标签列表，无法看到游戏B、游戏C的标签信息。

5. 游戏A运营创建、编辑自定义标签时，仅对游戏A生效，标签数据与其他游戏完全隔离，不会出现跨App标签混淆。

6. 系统管理员查看全平台标签列表时，可按App维度筛选标签，清晰区分不同游戏的标签数据，便于统筹管理。

7. 异常场景：权限分配错误，运营账号可看到无权限的App标签；标签数据隔离失效，出现跨App标签关联错误。

# 三、功能需求（详细版）

## 3.1 标签基础信息管理（核心功能）

### 3.1.1 功能描述

实现标签基础信息的全生命周期管理，包括标签的新增、编辑、启用、禁用、查询、删除，确保每款标签有唯一标识，信息准确可追溯，适配多游戏隔离需求，区分系统标签与自定义标签的操作权限。

### 3.1.2 核心字段定义（详细）

|字段名称|字段类型|是否必填|默认值|取值范围|字段说明|
|---|---|---|---|---|---|
|标签ID（TagID）|字符串|是|系统自动生成|长度16-32位，由字母、数字、下划线组成，唯一不重复|标签的唯一标识，用于标签关联、接口调用、数据隔离的核心关联字段，生成后不可修改。|
|标签名称|字符串|是|无|长度1-30位，不包含特殊字符，本App内唯一|标签的展示名称，用于前端展示、搜索、运营区分，系统标签名称不可修改，自定义标签可修改。|
|标签类型|单选|是|无|系统标签、自定义标签|系统标签：由系统按规则自动生成，不可手动创建、删除，仅可编辑描述、启用/禁用；自定义标签：由运营手动创建，可全生命周期操作。|
|标签分类ID|字符串|是|无|已存在的标签分类ID|关联的标签分类，用于标签归类管理，便于筛选查询，可修改。|
|AppID|字符串|是|当前操作App的ID|已存在的AppID|关联的游戏App，实现标签按App隔离，系统标签可关联单个或多个App（全局标签），自定义标签仅关联当前操作App。|
|标签描述|文本|否|无|长度0-200位|描述标签的用途、关联条件（如“高频登录用户：近7天登录≥5次”），便于运营理解和使用。|
|标签状态|单选|是|启用|启用、禁用|启用：标签可用于关联、圈选、推送；禁用：标签不可使用，关联关系保留，可重新启用。|
|创建人|字符串|是|当前登录用户|系统用户名|标签的创建用户，系统标签创建人为“系统”，不可修改。|
|创建时间|时间戳|是|系统当前时间|格式：yyyy-MM-dd HH:mm:ss|标签创建的时间，不可修改。|
|更新时间|时间戳|是|系统当前时间|格式：yyyy-MM-dd HH:mm:ss|标签信息最后一次更新的时间，修改标签信息后自动更新。|
|删除标识|布尔值|是|false|true（已删除）、false（未删除）|逻辑删除标识，删除标签时仅标记为true，不物理删除数据，便于追溯，系统标签不可删除。|
### 3.1.3 具体操作功能

- 新增标签：
        

- 系统标签：仅系统管理员可操作，点击“新增系统标签”，选择关联App（单个/多个）、标签分类，填写标签名称、描述，配置标签规则（详见3.3节），点击“保存”，系统校验标签名称唯一性、规则合法性，校验通过后生成系统标签。

- 自定义标签：游戏运营（本App）、系统管理员可操作，点击“新增自定义标签”，填写标签名称、描述，选择标签分类，点击“保存”，系统校验标签名称在本App内的唯一性，校验通过后生成自定义标签，默认状态为“启用”。

- 校验规则：标签名称不可为空、不可包含特殊字符，本App内不可重复；系统标签需配置有效规则，自定义标签需选择有效分类。

编辑标签：
        

- 系统标签：仅可编辑标签描述、标签分类、标签状态、关联App、标签规则，不可编辑标签ID、标签名称、创建人、创建时间，编辑后点击“保存”，系统校验规则合法性，校验通过后更新，同步更新标签关联数据（若规则修改）。

- 自定义标签：可编辑标签名称、描述、标签分类、标签状态，不可编辑标签ID、创建人、创建时间，编辑后点击“保存”，系统校验标签名称唯一性，校验通过后更新。

- 限制：“禁用”状态的标签可正常编辑，编辑后状态不变；已删除（逻辑删除）的标签不可编辑。

启用/禁用标签：
        

- 系统管理员可操作所有标签的启用/禁用；游戏运营仅可操作本App的自定义标签的启用/禁用。

- 点击“启用/禁用”按钮，系统弹出确认提示（如“确认禁用该标签？禁用后不可用于圈选、推送”），确认后切换标签状态。

- 禁用标签后，其关联关系保留，标签不再展示在圈选、推送的标签列表中；启用后，恢复展示和使用权限。

查询标签：
        

- 支持多条件筛选：按AppID（下拉筛选）、标签类型（下拉筛选）、标签分类（下拉筛选）、标签状态（下拉筛选）、标签名称（模糊搜索）、创建时间范围（日期选择）。

- 搜索结果展示：标签ID、标签名称、标签类型、标签分类、关联App、关联设备数、关联玩家数、状态、创建人、创建时间，支持分页（每页10/20/50条）。

- 权限控制：游戏运营仅能看到本App的标签；系统管理员可查看所有App的标签，可按App维度筛选。

- 详情查看：点击标签ID/名称，进入标签详情页，展示标签基础信息、关联规则（系统标签）、关联设备/玩家列表、操作日志。

删除标签：
        

- 系统标签：不可删除，仅可禁用。

- 自定义标签：游戏运营（本App）、系统管理员可操作，仅可删除“禁用”状态的自定义标签，删除后逻辑删除标识设为true，前端不再展示，数据保留在数据库中可追溯。

- 操作提示：点击“删除”，系统弹出确认提示“确认删除该标签？删除后不可恢复，关联关系将归档”，确认后完成删除。

## 3.2 标签分类管理（核心功能）

### 3.2.1 功能描述

实现标签分类的全生命周期管理，建立标准化的标签分类体系，支持按游戏App、标签用途（如设备标签、玩家行为标签、玩家偏好标签）进行分类，便于标签的归类、筛选和管理，适配多游戏隔离需求。
